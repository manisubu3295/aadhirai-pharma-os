- In the sales return / refund API (for example `POST /api/sales-returns` or whichever endpoint currently handles refunds/returns):

    - If `refund_mode` is CASH/UPI/CARD, reject the request with a 400 error and a clear message:
      `"Credit bill refunds must be processed as credit adjustments, not cash/card/UPI."`
  - For non-credit invoices, keep existing behaviour.

- Ensure that any accounting/collection entry for this refund:
  - Does NOT create a positive cash entry for credit refunds.
  - Instead, creates a credit adjustment entry that reduces the outstanding credit for that customer or the credit bucket (depending on how credits are currently tracked).

3) Frontend (New Sale / Refund UI):
- In any refund dialog/page where refund mode is selected:
  - When loading a refund for an invoice paid in CREDIT:
