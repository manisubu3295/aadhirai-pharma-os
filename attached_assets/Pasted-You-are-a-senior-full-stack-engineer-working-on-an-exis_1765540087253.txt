  - Not be allowed to be empty (no blank value).

- Add an endpoint to fetch a specific return for printing later, e.g.:
  - GET /api/sales-returns/:id/print or reuse an existing invoice-print endpoint that can also handle returns.
- Ensure the return invoice/credit note object is generated consistently and can be used to render HTML/PDF for printing.

Frontend:
- In the “Process Return” UI:
  - For each item row, the return quantity input should be initialized with value `0`.
  - The input must not be allowed to become empty; it should always contain a numeric value (0 or more).
    - Use Angular form validation and input type="number" with min="0".
    - Show an error if user tries to submit with invalid/empty quantity.
  - After a return is successfully processed:
    - Show a success message.
    - Open or provide a button to “Print Return Tax Invoice” (either in a new tab or via print dialog using the generated data).

  - Add a “Print Return” action to re-open the return invoice/credit note for printing at any time.

3) Prevent underpaid invoices (non-credit)
────────────────────────────────
Business rule:
- The system must NOT allow invoice generation if the received amount is less than the net amount when payment method is NOT credit.
- Exception: If the payment method is CREDIT, partial or zero payment at billing time is allowed (credit sale).

Backend:
- In the invoice creation endpoint (e.g. POST /api/invoices):
  - Read `payment_mode` and `amount_received` (or equivalent fields).
  - Calculate `net_amount` / total payable based on items, discount, GST, etc.
  - If `payment_mode` is CASH, UPI, or CARD (any non-credit mode):
    - If `amount_received < net_amount`, reject the request with 400 and error:
      "Received amount cannot be less than net amount for non-credit payments."
  - If `payment_mode` is CREDIT:
    - Allow invoice creation even if amount_received is 0 or less than net_amount.
- Ensure this validation does not break existing invoices already stored.

Frontend:
- In New Sale / Billing page:
  - When the user selects payment mode (Cash/UPI/Card/Credit) and enters received amount:
    - Optionally add client-side validation to warn the user if amount_received < net_amount for non-credit modes.
  - Ensure that on submit, any backend validation error from above is shown clearly in a toast/snackbar/dialog.
